## 核心架构：双层系统

### 为什么双层系统？

**问题：** Codex/Claude Code 单独用不够好

**原因：** 上下文窗口是固定的，你只能二选一：
- 塞满代码 → 没空间放业务上下文
- 塞满客户历史 → 没空间放代码

**结果：**
- 不知道功能是为哪个客户做的
- 不知道上次类似需求为什么失败了
- 不知道产品定位和设计原则

### 双层架构的优势

**类比：**
- Codex/Claude Code = 专业厨师，只管做菜
- OpenClaw = 主厨，知道客人口味、食材库存、菜单定位

**核心变化：**
- 从"管理 Claude Code"变成"管理一群 AI 管家"
- 这个管家再去管理一个 Claude Code

## 两层架构设计

### OpenClaw（编排层）职责
- 持有所有业务上下文（客户数据、会议记录、历史决策、成功/失败案例）
- 把业务上下文翻译成精确的 prompt
- 让这个 Agent 专注做它们擅长的事：写代码
- 根据任务类型选择合适的代理（Codex/Claude Code/Gemini）
- 监控所有代理的进度，失败了会分析原因并调整 prompt 重试
- 完成后通过 Telegram 通知作者

### Agent（执行层）职责
- 读写代码
- 运行测试和构建
- 提交代码和创建 PR
- 响应 code review 的反馈

**安全边界：**
- 执行层的 Agent 永远不会接触生产数据库
- 也不会看到客户的敏感信息
- 它们只拿到"完成这个任务需要知道的最小上下文"

## 完整工作流：8步从客户需求到PR合并

### 步骤1：客户需求理解
- 客户打电话，说希望能复用已配置好的设置
- **神奇之处：零解释成本**
- 因为所有会议记录自动同步到 Obsidian
- Zoe（OpenClaw）已经读过通话内容，知道客户是谁、业务场景、现有配置

### 步骤2：启动代理
- Zoe 为任务创建独立 git worktree（隔离环境）
- 启动 tmux 会话（让 Agent 后台运行）

```bash
# 创建 worktree + 启动代理
git worktree add ../feat-custom-templates -b feat/custom-templates origin/main
cd ../feat-custom-templates && pnpm install
tmux new-session -d -s "codex-templates" \
  -c "/path/to/repo" \
  "$HOME/.codex-agent/run-agent.sh templates gpt-5.3-codex high"
```

**为什么用 tmux？**
- 可以中途干预
- 如果 AI 走偏，不用杀掉重来，直接在 tmux 里发指令

### 步骤3：自动监控
- 每10分钟检查一次所有代理状态
- **重点：** 检查客观事实，不是"问"Agent 进度（那样很费token）
  - tmux 会话还活着吗？
  - 有没有创建 PR？
  - CI 状态如何？
  - 如果失败，是否需要重启（最多重试3次）

### 步骤4：Agent 创建 PR
- Agent 写完代码，提交，推送
- 用 `gh pr create --fill` 创建 PR

### 步骤5：自动 Code Review
每个 PR 会被三个 Agent 审查：
- **Codex Reviewer:** 擅长发现边界情况、逻辑错误、错误处理
- **Gemini Code Assist Reviewer:** 发现安全问题、扩展性问题
- **Claude Code Reviewer:** 基本没用（过度谨慎，大部分是过度设计）

### 步骤6：自动化测试
CI 管道运行：
- Lint + TypeScript 检查
- 单元测试
- E2E 测试
- Playwright 测试（生产环境一模一样的预览）

**新规则：** 如果 PR 改了 UI，必须在描述里附截图，否则 CI 直接失败
- 大幅缩短 review 时间

### 步骤7：人工 Review
作者收到 Telegram 通知："PR #341 准备好了，可以 review。"
- CI 全绿
- 三个 AI 审查者都批准
- 截图展示 UI 变化
- 所有边界情况都被 review

**作者只需 5-10 分钟**
- 很多 PR 甚至不看代码，只看截图就合并了

### 步骤8：合并
- PR 合并
- 每天 cron 清理孤立 worktree 和任务记录

**完整流程走完：** 客户需求 → 代码上线，可能只用了 1-2 小时
- 作者实际投入可能只有 10 分钟

## 让系统更聪明的机制

### 机制1：改进版 Ralph Loop - 动态学习

**问题：** 大多数实现的 prompt 是静态的

**解决方案：**
- Agent 失败时，Zoe 不会用同样的 prompt 重启
- 她带着完整的业务上下文，分析失败原因，然后 **重写 prompt**

**坏例子（静态 prompt）：**
```bash
"实现自定义模板功能？"
```

**好例子（动态调整）：**
```bash
"停一下。客户要的是 X，不是 Y。这是他们在会议里的原话：
'我们希望保存现有配置，而不是从头创建新的。
重点是配置复用，不要做新建流程。'"
```

**Zoe 能做这种调整：**
- 客户在会议里说了什么
- 这家公司是做什么的
- 上次类似需求为什么失败了

### 机制2：主动找活干

Zoe 不会等你分配任务，她会主动：

**早上：**
- 扫描 Sentry 发现 4 个新错误 → 启动 4 个 Agent 调查修复

**会议后：**
- 扫描会议记录发现 3 个客户提到的功能需求 → 启动 3 个 Codex

**晚上：**
- 扫描 git log → 启动 Claude Code 更新 changelog 和客户文档

**作者散步回来，Telegram 显示：**
- 7 个 PR 准备好了
- 3 个新功能
- 4 个 bug 修复

### 机制3：Agent 选择策略

**Codex (gpt-5.3-codex) - 主力**
- 后端逻辑、复杂 bug、多文件重构、跨代码库推理
- 慢但彻底，占 90% 任务

**Claude Code (claude-opus-4.5) - 速度型**
- 前端工作、git 操作、权限问题
- 速度快（作者以前更常用，但 Codex 5.3 出来后就换了）

**Gemini - 设计型**
- 有设计审美
- 漂亮 UI：先用 Gemini 生成 HTML/CSS 规范，再交给 Claude Code 实现
- Gemini 设计，Claude 构建

## 关键洞察

### 1. 一个人 = 一个开发团队
- Git 历史看起来像"刚招了一个开发团队"
- 实际上只有一个人

### 2. 成功的关键是上下文专业化
- 不是堆了一堆 AI 工具然后疯狂生成垃圾代码
- 而是从"管理 Claude Code"变成"管理一群 AI 管家"

### 3. 瓶颈是 RAM，不是 token 成本
- 每个 Agent 需要自己的 worktree + node_modules
- 5 个 Agent = 5 个并行 TypeScript 编译
- Mac Mini (16GB RAM) 最多同时跑 4-5 个 Agent
- Mac Studio M4 Max (128GB RAM) = $3500
- 作者买了一台 Mac Studio M4 Max

### 4. 10 分钟从零到运行
**最简单的方法：**
把整篇文章复制到 OpenClaw，告诉它：
"按照这个架构，给我的代码库实现一套 Agent 集群系统。"

然后它就会：
- 读取架构设计
- 创建脚本
- 设置目录结构
- 配置 cron 监控

**10 分钟搞定。**
